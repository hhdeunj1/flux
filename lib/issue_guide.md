# GitHub 이슈 자동 생성 가이드

> Claude가 이슈를 자동 생성할 때 따르는 규칙 모음

---

## 1. 공통 규칙

### 제목
- 반드시 `[자동생성🤖]` 태그를 제목 앞에 추가 (모든 레포 공통)
  - 사용자가 직접 편집하므로 Claude는 항상 달아줄 것
  - 예: `[자동생성🤖] [간편가입 웹] v4.11 신규 회원 가입 대상 확대`
- 레포 식별 태그(`[드앱]`, `[택시앱]` 등)는 제목에 불필요 — 레포 자체가 맥락

### 본문 구조화
- 테이블/구조로 정리하면 읽기 좋은 내용은 **처음부터 반영** (나중에 제안하지 말 것)
- 조건 분기표, 우선순위표, 케이스별 매핑 등 → 처음부터 테이블로 작성

## 2. Feature 이슈 작성 원칙

### Feature 섹션 구조

```markdown
## 배경
...

## 개요
한줄 요약 문장 (변경 내용 한 줄 설명)

> [진입 경로] 온보딩 > 개인/법인 선택 > 회원정보 입력
> [연관 화면] [프레임명](https://www.figma.com/design/{fileKey}/driver_taxi?node-id={nodeId})
> [연관 MD] [MD파일명 > 섹션 경로](앵커 링크)

## Spec [[Figma>]](기획 Figma URL)

### AS-IS
...

### TO-BE
...

---

<details>
<summary>🤖 AI Comments</summary>
...
</details>
```

### Feature 작성 주의사항

- `## 개요`: 한줄 요약(blockquote 없이 일반 텍스트) + 진입경로/연관화면/연관MD
  - 연관 화면: 디자인 Figma 참고 (해당 Product 디자인 Figma를 참고)
  - 연관 MD: 기획 md 참고 (해당 Product md를 중심으로 참고)

- `## Spec` 헤더에 Figma 기획 프레임 링크 추가:
  - 해당 Product의 기획 Figma에서 `#이슈번호` 이름의 프레임 탐색 → node-id 추출 → URL 생성
    - 프레임 있을 때: `## Spec [[Figma>]](기획 Figma의 #이슈번호 프레임 URL)`
    - 프레임 없을 때: `## Spec [Figma>](프레임 생성 시 자동 연결)` (링크 없이 텍스트만)

- `## AS-IS`, `## TO-BE`
  - TO-BE와 대응되는 AS-IS 문장을 각각 먼저 쓰고, 변경 핵심 단어를 **볼드** 처리
    - 예: "개인 심사 등록 항목에 나이 **없음**" / "개인 심사 등록 항목에 나이 **추가**"

- `## AS-IS`
  - 같은 레포 내 MD를 참조해서 현재 상태를 구체적으로 작성 + 출처 링크 첨부
    - 표기 형식: `> [참고] [MD파일명 > 섹션 경로](앵커 링크)`
    - 링크는 MD 파일이 아닌 해당 **섹션 앵커**로 직접 이동하는 링크

- `## TO-BE`
  - 기존 항목 전체를 동일하게 나열하고, 추가/변경 항목을 더해서 작성 (전후 비교 가능하게)
    - 항목명은 AS-IS와 동일하게 일반 텍스트로
    - 변경 구분은 항목명 뒤에 `` `추가` ``, `` `변경` ``, `` `삭제` `` 레이블로 표시
  - 버튼 동작 하위 불렛은 **새로 추가된 항목에만** 작성 (기존 항목은 단순 나열)

- `## AI Comments`: 반드시 `<details>` 블록으로 접힌 상태로 노출
  - 이슈 관련 MD 섹션을 읽고, 스펙 변경으로 인해 같이 정의되어야 할 연관 영역을 찾아 추가하되 중요도를 레이블로 표기
  - **찾아야 할 것**
    - 동일 화면/기능을 참조하는 다른 섹션
    - 세션·데이터 보관 등 정책이 이 변경에 영향 받는 경우
    - 개인/법인 등 케이스 분기가 있는데 이슈에서 한쪽만 정의된 경우
    - 관련 화면(반려·만료·철회 등)에서도 동일하게 적용해야 할 규칙
  - **중요도 레이블** 표기 (중요도 높고 확실한 순으로 정렬)
    - `필수`: 미정의 시 개발 불가 또는 버그 확실
    - `권장`: 엣지케이스 발생 가능
    - `참고`: 추후 이슈화 고려
  - 연관 MD는 섹션 앵커 직링크로 표기

## 3. Task 이슈 작성 원칙

### Task 섹션 구조

```markdown
## 배경
...

## TO-DO
- [ ] 작업 항목 1
- [ ] 작업 항목 2
```

### Task 작성 주의사항

- `## 배경`: "현재 문서 문제점" 관점이 아니라 **"왜 지금 이 작업이 필요한가"** 관점으로 작성
  - ❌ "MD에 분산 기술되어 혼선 발생 가능"
  - ✅ "기능이 추가되며 케이스가 많아져 정리 필요"

- `## TO-DO`: 사용자의 **실제 작업 흐름** 순서로 구성 (예: Figma 먼저 → MD 반영)
  - 세부 항목을 Claude가 추측해서 과하게 쪼개지 말 것
  - 작업 단위는 본인이 할 일 덩어리 기준

## 4. Research 이슈 작성 원칙

### Research 섹션 구조

```markdown
## 배경 및 목표
- 배경:
- 목표:

## 계획
- 방법:
- 일정:
- 질문지 (인터뷰의 경우)

## 결과
- 응답 결과 (Raw)

## 인사이트 및 Action Item
**인사이트 1**
- 내용
- Action Item

**인사이트 2**
- 내용
- Action Item
```

### Research 작성 주의사항

- 제목 형식: `[YY년M월] [인터뷰]/[서베이]/[데이터분석] 리서치명`
- 이슈 생성 시점에는 배경/목표/계획만 채우고, 결과·인사이트는 리서치 완료 후 업데이트
- 질문지는 인터뷰 방식에만 해당, 서베이·데이터분석은 생략 가능

## 5. Milestone 이슈 작성 원칙

### Milestone 섹션 구조

```markdown
## 목표
- 목표 1
- 목표 2

## 일정 [[셔클 제품 일정>]](https://docs.google.com/spreadsheets/d/1b3G6gC7I6tu8eXVUmM6AVj6nVLdwKdp7XWpjwpq8xbw/edit?gid=1742115418#gid=1742115418)
- 릴리즈 목표:
- 상세 일정
  - [ ] 기획 리뷰:
  - [ ] spec fix:
  - [ ] md 작성:
  - [ ] dev. sanity:

## Spec [[Figma >]](figmaURL)
**목표 1**
- 하위 이슈 1 (GitHub URL)
- 하위 이슈 2 (GitHub URL)

**목표 2**
- 하위 이슈 1 (GitHub URL)
- 하위 이슈 2 (GitHub URL)
```

### Milestone 작성 주의사항

- 제목 형식: `[드라이버앱 4.n] 마일스톤명` (레포 템플릿 형식 준수)
- label: `Epic`
- 이슈 생성 시점에는 목표·일정 중심으로 채우고, Spec 하위 이슈는 생성되는 대로 추가
- 일정 셔클 제품 일정 시트 링크는 고정값으로 항상 포함

## 6. 이슈 테이블 구조 패턴

케이스가 복잡하거나 화면/항목 간 매핑이 필요할 때 아래 두 테이블로 분리:

1. **케이스별 랜딩 화면 테이블**: 단계 × 케이스 → 결과 화면 매핑
2. **화면 구성 테이블**: 각 화면의 title / sub title / 버튼 스펙

- 두 테이블은 `[A]~[Z]` 알파벳 식별자로 연결
  - 랜딩 화면 칼럼에 `[A] 화면명`, 화면 구성 첫 컬럼 `#` 열에 같은 알파벳 표기
- **화면 이름 부여 기준**: title + sub title + 버튼이 모두 같아야 같은 화면. 하나라도 다르면 별도 이름

## 7. 이슈 생성 후 체크리스트

Feature 이슈에서 아래 항목이 빠져 있으면 개선 제안으로 알려주기:
- 입력 방식 명세 (직접입력 / 자동계산 / 선택 등)
- 항목 표시 위치/순서
- 유효성 검사 조건 (범위, 필수 여부 등)
- Acceptance Criteria
